一、引言

1.1 项目背景

唐诗是中国古典文学的瑰宝，其中蕴含丰富的意象资源。传统文学研究依赖人工标注和分析，效率较低且难以量化。"唐诗意象智能分析系统"旨在通过计算机技术，对唐诗中的意象进行结构化标注、多维统计分析和智能检索，为文学研究者提供数字化工具支持。

1.2 软件概述

本系统是一个基于 Flask 框架的 Web 应用程序，集成了以下核心能力：

- 意象数据管理：对 313 首唐诗中的 12,738 个分析单元进行结构化存储与管理，其中标注为"意象"的条目共 4,896 条
- 多维度统计分析：支持意象频次、分类域、情感倾向、感知通道等 15 个维度的统计与可视化
- RAG 智能问答：基于向量检索（ChromaDB）+ 大语言模型，实现唐诗相关知识的检索增强生成
- 认知诗学解析：利用 AI 对意象进行感知层、文化层、情感层、结构层、跨诗比较的多维解析
- 数据导出：支持 CSV、JSON、分析报告等多种格式的数据导出

三、数据结构与数据库设计

3.1 数据源

系统数据来源于 poem_json/tang_poetry_313.json 文件（10.9 MB），包含 313 首唐诗的结构化标注数据。

每条诗歌记录包含以下结构：

诗歌
  +-- 诗歌编号 (string)
  +-- 标题 (string)
  +-- 作者 (string)
  +-- 朝代 (string)
  +-- 分类标签 (string)
  +-- 体裁 (string)
  +-- 原文 (string)
  +-- 诗行 (array)
  |     +-- 诗行对象: {诗行编号, 原文, 注释}
  +-- 分析单元 (array)  <- 核心标注数据
  |     +-- 分析单元对象:
  |           - 单元编号, 诗行编号, 文本, 行内位置
  |           - 词性, 成分类型, 是否意象
  |           - 大类编码, 子类编码
  |           - 感知通道, 素材类型, 内部结构
  |           - 指涉来源, 表现功能
  |           - 文化流通性, 跨文化性, 认知强度
  |           - 核心意象, 结构功能组
  |           - 情感极性, 情感类别, 情感置信度
  +-- 情感轨迹 (array)
  +-- 意象关系 (array)

三大类（大类编码）：

编码  名称       子类
1     自然意象   1-1 天文意象、1-2 地理意象、1-3 植物意象、1-4 动物意象
2     社会意象   2-1 生产生活意象、2-2 军事战争意象、2-3 制度观念意象
3     人文意象   3-1 人造器物意象、3-2 人类自身意象、3-3 人物角色意象、3-4 文化意象

3.4 数据模型类（models.py）

系统使用 Python dataclass 定义了完整的数据模型：

PoemLine（诗行）：诗行编号、原文、注释
AnalysisUnit（分析单元）：25 个字段覆盖文本、词法、感知、情感等维度
EmotionTrajectory（情感轨迹）：诗行编号、诗行原文、平均情感极性、主导情感、情感波动值
ImageRelation（意象关系）：关系编号、来源/目标文本、关系类型、关系强度
Poem（诗歌）：包含上述所有子对象的完整诗歌模型
TracebackItem（溯源条目）：用于前端展示的扁平化溯源表条目
AnalysisResult（解析结果）：AI 解析会话的状态封装

3.5 向量数据库设计

基于 ChromaDB 的持久化向量数据库，存储在 rag_db/ 目录：

集合名称：poems
距离函数：cosine（余弦相似度）
向量模型：text-embedding-qwen3-embedding-4b
文档内容：诗歌原文全文
元数据：标题、作者、诗歌编号
检索方式：支持按作者过滤 + 语义向量检索
Top-K 默认值：5


四、功能模块详细说明

4.1 数据加载与预处理模块

文件：app.py（build_traceback_dataset 函数）+ preprocessor.py

功能描述：

从 poem_json/ 目录扫描 JSON 数据文件，执行以下处理流程：

1. 文件扫描：搜索目录下所有 .json 和 .txt 文件，自动跳过 all_data.json 等非数据文件
2. JSON 清洗：使用正则表达式去除 Markdown 代码包裹（```json、```）
3. 诗歌提取：递归遍历 JSON 结构，查找包含"分析单元"数组的诗歌节点
4. 去重机制：基于"标题_首句"指纹进行诗歌级去重
5. 溯源表构建：遍历每个分析单元，筛选是否意象 = "1" 的条目，补充分类名称映射、情感信息、感知维度等
6. 缓存策略：使用全局变量 _DATA_CACHE 缓存解析结果，支持通过 API 手动刷新

预处理工具（preprocessor.py）：
- clean_json_content()：清理 JSON 中的不规范字符和 Markdown 包裹
- validate_poem_json()：校验诗歌 JSON 结构的完整性
- parse_json_file()：安全解析单文件，返回数据和错误列表
- batch_validate()：批量校验目录下所有 JSON 文件
- backup_data()：备份数据目录
- convert_raw_poem_to_model()：将原始字典转换为 Poem 模型对象
- extract_image_statistics()：从 Poem 模型列表提取意象统计摘要

涉及的数据变量：

溯源表 (traceback) = List[Dict]
  - poem_id: 诗歌编号
  - title: 诗歌标题
  - author: 作者
  - genre: 体裁/分类标签
  - cat: 分类域名称（中文）
  - txt: 意象文本
  - line: 所在诗句原文
  - emotion: 情感描述
  - dimensions: 四层摘要（感知|素材|指涉|表现）
  - 及其他 20 个原始标注字段

4.5 AI 智能问答模块（RAG）

文件：query_rag.py（237 行）+ app.py 中的 /api/ask 路由

功能描述：

基于检索增强生成（RAG）架构的唐诗知识问答系统：

检索流程：
1. 意图提取：从用户问题中检测诗人姓名（基于 25 位已知诗人列表），提取语义查询关键词
2. 向量化：调用嵌入模型 API 将问题转为向量
3. 向量检索：在 ChromaDB 中检索 Top-K 相似诗歌（支持按作者过滤）
4. 上下文构建：将检索结果拼接为包含标题、作者、原文的上下文

生成流程：
5. 消息构建：首轮构建包含系统角色、检索上下文、用户问题的消息列表；多轮追问直接拼接历史
6. 流式调用：通过 SSE (Server-Sent Events) 流式传输大模型回答
7. 结果展示：前端分两栏展示——左栏检索到的诗歌卡片（可展开全文），右栏流式回答

多轮对话：
- 保存对话历史（role/content 结构）
- 追问时携带完整历史记录
- 重新检索仅在首轮进行，追问直接使用历史对话

前端交互（templates/index.html AI 问答 Tab）：
- 输入框 + 检索按钮
- 检索结果诗歌卡片（展示标题、作者、相似度、全文预览）
- 流式回答区（实时追加文本）
- 多轮追问输入框
- 错误提示

4.6 认知诗学意象解析模块

文件：app.py（build_analyze_system_prompt + /api/analyze 路由）+ query_rag.py（stream_cognitive_analysis）

功能描述：

对单条意象数据进行深度认知诗学解析，采用"右侧抽屉"交互模式：

解析维度（系统提示词）：
1. 感知层：触发的感官体验和通感效果
2. 文化层：典故来源和文化符号意义
3. 情感层：基于情感极性分析情感功能
4. 结构层：作为特定结构功能组在诗中的结构作用
5. 跨诗比较：此意象在其他诗歌中的异同

系统提示词构建（build_analyze_system_prompt）：
- 动态注入当前意象的结构化字段数据（约 20 个字段）
- 指令要求严格基于提供的数据分析，不编造

前端抽屉交互：
- 点击"解析"按钮从右侧滑入抽屉
- 抽屉可拖拽调整宽度（320px 至视口宽度减 80px）
- 展示意象元信息（文本、诗歌名称、诗句）
- 标签 chips 展示感知通道、分类域、表现功能、认知强度、情感等信息
- 对话式消息列表，支持 Markdown 渲染
- 首轮自动发送默认解析请求
- 支持多轮追问
- 流式渲染解析内容（实时更新气泡）
- 缓存机制：同一意象的解析结果缓存，重复打开不重复请求
- 中断处理：支持 AbortController 取消请求

功能描述：

实现两级缓存体系，均支持可配置的 TTL：

MemoryCache（内存缓存）：

- 基于字典存储，支持 TTL 过期自动清除
- 全局单例：memory_cache
- 方法：get, set, delete, clear, size, keys

FileCache（文件缓存）：
- 持久化到磁盘 cache/ 目录
- 每个缓存项存储为独立 JSON 文件
- 线程安全
- 全局单例：file_cache
- 方法：get, set, delete, clear, size

cached 装饰器：
- 支持装饰函数自动缓存返回值
- 缓存键生成规则：{module}:{func_name}:{args}:{sorted_kwargs}
- 支持选择缓存后端（memory/file）
- 全局开关：CACHE_ENABLED
